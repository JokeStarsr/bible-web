# 系统架构方案

## 1. 文档目标

本文档基于 `README.md` 的产品需求，给出该基督徒线上 Web 服务的推荐系统架构方案，用于指导后续数据库设计、API 设计、前后端实现和部署落地。

## 2. 架构目标

系统架构需要满足以下目标：

1. 支持账号注册、登录、密码找回等完整鉴权链路。
2. 支持随机经文生成、解经、标注、感悟、交流、私信和赞美播放。
3. 支持后续扩展多译本、收藏、每日灵修、主题经文推荐等功能。
4. 支持 AI 解经生成或结构化资料融合输出。
5. 支持移动端与桌面端的良好体验。
6. 支持内容审核、举报、私密数据保护和基础运营管理。

## 3. 总体架构

推荐采用前后端分离架构，并保留 AI 服务与后台管理扩展位。

### 3.1 推荐分层

1. Web 前端层
2. API 网关层或统一后端入口层
3. 业务服务层
4. AI 解经服务层
5. 数据存储层
6. 文件与媒体资源层
7. 内容审核与管理层

### 3.2 推荐技术路线

#### 前端

1. React + Next.js
2. TypeScript
3. Tailwind CSS 或组件库
4. 富文本或结构化编辑组件，用于标注与感悟输入

#### 后端

1. Node.js
2. NestJS，优先推荐
3. REST API，首版优先
4. JWT + Refresh Token 或基于 Session 的鉴权方案

#### 数据层

1. PostgreSQL，优先推荐
2. Redis，用于验证码、限流、缓存、会话辅助
3. 对象存储，用于音频资源、头像、静态内容等

#### AI 与内容生成

1. 独立的 Exegesis 生成模块
2. 支持调用外部大模型 API
3. 支持结构化知识资料预处理与检索增强

## 4. 模块划分

建议按领域拆分为以下核心模块。

### 4.1 Auth 认证模块

职责：

1. 注册
2. 登录
3. 退出登录
4. 邮箱验证码
5. 找回密码
6. 重置密码
7. 修改密码
8. 登录风控和限流

### 4.2 User 用户模块

职责：

1. 维护用户基础资料
2. 头像、昵称、简介等扩展字段
3. 用户状态管理
4. 用户隐私设置

### 4.3 Bible Scripture 经文模块

职责：

1. 圣经卷章节数据管理
2. 随机经文生成
3. 连续节段拼接
4. 整章查询
5. 经文出处格式化

### 4.4 Exegesis 解经模块

职责：

1. 根据当前经文生成或返回解经内容
2. 按固定结构组织输出内容
3. 对接 AI 服务或结构化资料库
4. 记录生成结果与缓存策略

### 4.5 Annotation 标注模块

职责：

1. 划线
2. 高亮颜色管理
3. 选区标注解释
4. 编辑、删除、查询个人标注

### 4.6 Reflection 感悟模块

职责：

1. 保存灵修感悟
2. 支持草稿自动保存
3. 私有 / 公开状态管理
4. 感悟浏览与详情页

### 4.7 Discussion 交流模块

职责：

1. 公开感悟评论
2. 多轮交流记录
3. 私信触发条件统计
4. 点赞、举报等扩展能力

### 4.8 Messaging 私信模块

职责：

1. 建立私信会话
2. 私信消息发送与读取
3. 会话状态控制
4. 屏蔽、关闭、举报

### 4.9 Praise 赞美播放模块

职责：

1. 随机赞美资源选择
2. 返回音频元数据与播放地址
3. 播放记录与偏好支持

### 4.10 Moderation 内容审核模块

职责：

1. 感悟内容审核
2. 评论内容审核
3. 举报处理
4. 敏感内容标记

### 4.11 Admin 后台管理模块

职责：

1. 用户管理
2. 内容管理
3. 举报管理
4. 赞美资源管理
5. 解经来源管理
6. 审计日志查看

## 5. 推荐的系统交互关系

### 5.1 主链路一：账号系统

1. 前端提交注册或登录请求到后端 API。
2. 后端 Auth 模块校验输入、写入用户凭证、返回登录态。
3. Redis 存储验证码、限流状态和临时安全令牌。
4. PostgreSQL 保存用户和认证相关数据。

### 5.2 主链路二：随机经文生成

1. 用户在前端选择生成 1 节、7 节、12 节、27 节、39 节或整章。
2. 后端 Scripture 模块根据规则生成经文范围。
3. 从圣经数据表中读取经文内容。
4. 返回结构化经文段落、出处和展示信息。

### 5.3 主链路三：解经

1. 用户点击“解经”按钮。
2. 前端调用 Exegesis API。
3. 后端先查询是否已有缓存或已审核内容。
4. 若无结果，则调用 AI 解经服务或知识融合逻辑生成内容。
5. 返回结构化解经内容给前端。
6. 可选择将生成结果缓存，减少重复生成成本。

### 5.4 主链路四：划线、标注和感悟

1. 用户在经文卡片中选中文本。
2. 前端生成选区定位数据和内容片段。
3. 后端保存高亮、颜色、备注和感悟内容。
4. 私有内容仅本人查询可见。
5. 公开内容可进入公开展示与交流流转。

### 5.5 主链路五：公开交流与私信触发

1. 用户在公开感悟下发表评论。
2. 后端记录评论并更新互动统计。
3. 系统根据规则判断是否达到私信触发条件。
4. 达标后创建可发起私信的权限记录。
5. 双方确认后进入 Messaging 模块进行私信交流。

### 5.6 主链路六：随机赞美播放

1. 前端点击随机赞美。
2. 后端 Praise 模块从资源表中随机抽取一个可用资源。
3. 返回音频标题、作者、封面、播放地址、循环支持信息。
4. 前端播放器负责播放、暂停、循环与 UI 控件。

## 6. 推荐的前端架构

### 6.1 页面层

建议拆分以下页面：

1. 首页
2. 注册页
3. 登录页
4. 找回密码页
5. 用户灵修首页
6. 经文解经详情区或阅读页
7. 公开感悟详情页
8. 私信页
9. 个人中心页
10. 管理后台页

### 6.2 前端功能组件

建议拆分以下组件：

1. 经文卡片组件
2. 随机生成按钮组组件
3. “解经”主按钮组件
4. 解经内容展示组件
5. 经文选区高亮组件
6. 感悟编辑器组件
7. 评论列表组件
8. 私信会话组件
9. 赞美播放器组件
10. 提示与反馈组件

### 6.3 状态管理建议

1. 用户登录态集中管理
2. 当前经文上下文集中管理
3. 当前解经数据单独缓存
4. 标注与感悟编辑状态本地临时保存
5. 播放器状态独立维护，避免切换页面时中断

## 7. 推荐的后端架构

### 7.1 分层建议

1. Controller 层：接收请求和返回响应
2. Service 层：封装业务逻辑
3. Repository / ORM 层：处理数据库交互
4. Integration 层：封装 AI、邮件、存储、审核等外部依赖
5. Guard / Middleware 层：做鉴权、审计、限流、日志记录

### 7.2 关键后端能力

1. 统一错误处理
2. 统一鉴权中间件
3. 请求参数校验
4. 操作审计日志
5. 统一分页结构
6. 内容状态流转控制

## 8. AI 解经模块设计建议

### 8.1 输出结构

AI 输出必须为固定结构，建议如下：

1. `summary`
2. `historicalBackground`
3. `writingBackground`
4. `contextAnalysis`
5. `keywordAnalysis`
6. `canonicalPosition`
7. `theologicalTheme`
8. `truthForPeople`
9. `practicalApplication`
10. `referenceSources`

### 8.2 生成策略

1. 优先查询缓存或已审核内容。
2. 没有缓存时，再调用大模型生成。
3. 若采用检索增强，先查资料库，再拼接生成上下文。
4. 保存生成版本与模型参数，便于后续质量追踪。

### 8.3 风险控制

1. 避免断章取义。
2. 避免输出极端神学结论。
3. 对资料来源和可信度进行标注。
4. 对争议内容增加提示语。

## 9. 内容审核与安全架构

### 9.1 需要审核的内容

1. 公开感悟
2. 评论回复
3. 私信内容，首版可先做举报后人工处理
4. 用户昵称与资料

### 9.2 审核策略

1. 发布前敏感词校验
2. 发布后举报处理
3. 管理员后台人工审核
4. 严重违规内容自动隐藏

### 9.3 安全要求

1. 密码哈希存储
2. 刷验证码和登录接口限流
3. 私有资源鉴权校验
4. 评论和私信防刷屏
5. 接口输入防注入、防越权

## 10. 部署架构建议

### 10.1 MVP 阶段

建议采用较轻量的部署方式：

1. 一个前端应用
2. 一个后端服务
3. 一个 PostgreSQL
4. 一个 Redis
5. 一个对象存储

### 10.2 后续扩展阶段

可逐步演进为：

1. 前端独立部署
2. API 服务独立扩容
3. AI 解经服务独立扩容
4. 审核服务和后台服务独立化

## 11. 关键技术决策建议

### 11.1 首版必须尽早确认

1. 圣经版本与版权来源
2. 赞美资源来源与版权
3. “20 轮交流”的准确算法
4. 私信权限是永久还是限时
5. AI 解经是否先人工审核后公开

### 11.2 开发策略建议

1. 先打通最短主链路，不要一开始就铺开全部模块。
2. 先让用户能注册登录并成功生成经文。
3. 再接解经与阅读体验。
4. 再做标注、感悟与互动。
5. 最后做后台、审核和性能优化。

## 12. 架构验收标准

1. 模块边界清晰，可独立开发。
2. 接口职责清楚，适合前后端并行。
3. 数据模型能覆盖 MVP 全流程。
4. AI 解经能力有明确输入输出结构。
5. 公共交流、私信、审核之间无明显职责冲突。
6. 系统具备后续扩展空间。
