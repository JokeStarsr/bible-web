# 执行顺序与里程碑文档

## 1. 文档目标

本文档将 `README.md` 第 17 节中的执行要求转化为清晰的实施路线图，帮助后续 AI 助理或开发团队按阶段推进，而不是无序并行开发。

## 2. 总体推进原则

1. 先确认架构与数据模型，再进入编码。
2. 先打通核心主链路，再补充扩展功能。
3. 每个阶段都要有明确交付物和完成标准。
4. 对需求中尚未确认的边界，必须在进入相关开发前定稿。

## 3. 阶段路线图

### 3.1 阶段 0：需求冻结与关键决策

目标：

1. 复核 `README.md`
2. 确认关键未定事项

必须确认的事项：

1. 圣经版本与版权来源
2. 解经内容来源策略
3. “20 轮交流”的计数规则
4. 私信解锁是永久、限时还是一次性
5. 赞美资源来源与版权
6. 首版后台范围

输出物：

1. 更新后的 `README.md`
2. 若有变更，同步修订 `docs` 目录文档

### 3.2 阶段 1：系统设计阶段

目标：

1. 完成系统架构设计
2. 完成数据库设计
3. 完成 API 设计

输入：

1. `README.md`

输出：

1. `docs/01-system-architecture.md`
2. `docs/02-database-design.md`
3. `docs/03-api-spec.md`

完成标准：

1. 模块划分明确
2. 数据模型可覆盖 MVP
3. 接口清单可支持前后端并行开发

### 3.3 阶段 2：项目初始化阶段

目标：

1. 初始化前后端工程
2. 建立数据库迁移和基础配置
3. 接入 Redis、邮件、鉴权基础能力

输入：

1. `docs/01-system-architecture.md`
2. `docs/02-database-design.md`

输出：

1. 前端项目骨架
2. 后端项目骨架
3. 数据库迁移脚本
4. 环境变量模板

完成标准：

1. 项目可本地启动
2. 基础服务可连通

### 3.4 阶段 3：核心链路一

目标：

优先实现：

1. 账号系统
2. 随机经文生成
3. 解经展示

原因：

1. 这是产品最短可验证价值链
2. 用户必须先登录，才能沉淀后续灵修数据
3. 经文 + 解经是产品最核心差异点

输出：

1. 可注册登录
2. 可随机生成经文
3. 可点击“解经”看到结构化内容

完成标准：

1. 用户从注册到解经的主链路全通
2. 桌面端和移动端都有基本可用体验

### 3.5 阶段 4：核心链路二

目标：

实现：

1. 划线标注
2. 感悟草稿与发布
3. 私有 / 公开可见性控制

输出：

1. 用户可对经文做个人灵修沉淀
2. 用户可公开发布感悟

完成标准：

1. 私有内容权限正确
2. 感悟内容可保存、可恢复、可发布

### 3.6 阶段 5：互动能力阶段

目标：

实现：

1. 公开感悟评论交流
2. 交流轮次统计
3. 私信解锁与私信会话
4. 随机赞美播放

说明：

1. 虽然赞美播放与交流不强耦合，但都属于第二层体验增强能力，可并入同一阶段推进

完成标准：

1. 公开感悟可形成互动
2. 达标后可进入私信
3. 随机赞美可正常播放

### 3.7 阶段 6：治理与打磨阶段

目标：

实现：

1. 举报能力
2. 管理后台基础能力
3. 内容审核能力
4. 性能优化
5. 视觉打磨

完成标准：

1. 平台具备基础治理能力
2. 核心页面体验明显优化

## 4. 推荐里程碑

### M1 设计完成

包含：

1. 架构文档完成
2. 数据库文档完成
3. API 文档完成
4. MVP 任务拆分完成

### M2 最小主链路可运行

包含：

1. 注册 / 登录完成
2. 随机经文完成
3. 解经展示完成

### M3 灵修沉淀能力完成

包含：

1. 划线
2. 标注
3. 感悟保存
4. 公开发布

### M4 互动闭环完成

包含：

1. 评论交流
2. 私信解锁
3. 私信收发
4. 随机赞美播放

### M5 首版可上线

包含：

1. 后台基础治理
2. 安全与性能优化
3. 关键页面视觉打磨

## 5. 风险与前置约束

### 5.1 数据版权风险

1. 圣经文本版权必须优先确认
2. 赞美资源版权必须优先确认

### 5.2 AI 内容风险

1. 解经内容可能存在神学争议或断章取义问题
2. 需要预留审核与来源标注机制

### 5.3 业务规则风险

1. “20 轮交流”定义不清会直接影响数据库、接口和统计逻辑
2. 私信解锁的有效期如果未确认，会影响会话权限模型

### 5.4 体验风险

1. 经文阅读和解经阅读是长文本场景，前端排版需要重点投入
2. “解经”按钮要醒目，但不能破坏整体灵修氛围

## 6. 开发协作建议

### 6.1 若由多人或多 AI 协作

推荐分工：

1. 一组负责认证与用户
2. 一组负责经文与解经
3. 一组负责标注、感悟、评论、私信
4. 一组负责前端 UI 与播放器

### 6.2 若由单个 AI 助理连续推进

推荐顺序：

1. 先做后端数据与接口骨架
2. 再做前端基础页面
3. 先打通账号
4. 再打通经文和解经
5. 再做灵修与互动
6. 最后做后台和优化

## 7. 每阶段交付检查清单

### 7.1 设计阶段检查

1. 是否已有明确架构图思路
2. 是否已有完整数据模型
3. 是否已有 API 清单

### 7.2 开发阶段检查

1. 是否已完成数据库迁移
2. 是否已完成接口鉴权
3. 是否已完成主要页面联调

### 7.3 上线前检查

1. 私有内容是否严格隔离
2. 公开内容是否具备举报与处理能力
3. 核心页面是否完成移动端适配
4. 音频和文本资源是否具备版权确认
