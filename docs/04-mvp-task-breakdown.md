# MVP 开发任务拆分文档

## 1. 文档目标

本文档将 `README.md` 的 MVP 范围拆解成可执行开发任务，供后续 AI 助理按优先级逐项实现、提交和验收。

## 2. 拆分原则

1. 先打通最短主链路，再补增强能力。
2. 每个任务尽量做到“可实现、可测试、可验收”。
3. 任务之间按依赖关系排序，避免无效并行。

## 3. 任务阶段总览

### 阶段 A：项目基础设施

1. 初始化前端项目
2. 初始化后端项目
3. 初始化数据库与迁移工具
4. 建立基础目录结构与代码规范
5. 建立环境变量和配置体系

### 阶段 B：账号系统

1. 注册
2. 登录
3. 找回密码
4. 修改密码
5. 登录态管理

### 阶段 C：经文主链路

1. 圣经数据导入
2. 随机经文生成
3. 经文展示 UI
4. 解经按钮与解经内容展示

### 阶段 D：个人灵修能力

1. 划线
2. 标注
3. 感悟草稿
4. 感悟发布

### 阶段 E：公共交流能力

1. 公开感悟流
2. 评论交流
3. 轮次统计
4. 私信解锁

### 阶段 F：赞美与平台治理

1. 随机赞美播放
2. 举报能力
3. 管理后台基础能力

## 4. 详细任务拆分

### 4.1 阶段 A：项目基础设施

#### A-01 初始化前端项目

目标：

1. 建立 React + Next.js + TypeScript 工程
2. 配置 Tailwind CSS
3. 配置基础页面路由

验收标准：

1. 项目可本地启动
2. 有首页、登录页、注册页空壳
3. 有统一布局与全局样式

#### A-02 初始化后端项目

目标：

1. 建立 NestJS 项目
2. 配置统一异常处理
3. 配置参数校验和基础日志

验收标准：

1. 后端可本地启动
2. 提供健康检查接口
3. 已建立模块化目录结构

#### A-03 初始化数据库与迁移工具

目标：

1. 连接 PostgreSQL
2. 接入 ORM
3. 建立迁移脚本执行流程

验收标准：

1. 本地可执行迁移
2. 数据库连接可通过环境变量配置

#### A-04 配置 Redis 与邮件能力

目标：

1. 配置 Redis 客户端
2. 配置邮箱验证码发送能力

验收标准：

1. 可写入和读取验证码缓存
2. 可调用邮件发送服务或模拟器

### 4.2 阶段 B：账号系统

#### B-01 实现用户注册

范围：

1. 发送邮箱验证码
2. 注册表单提交
3. 用户与凭证写入数据库

依赖：

1. A-02
2. A-03
3. A-04

验收标准：

1. 新用户可完成注册
2. 用户名和邮箱唯一校验生效
3. 密码按安全规则校验

#### B-02 实现用户登录与退出

范围：

1. 邮箱密码登录
2. Access Token / Refresh Token
3. 退出登录

验收标准：

1. 登录成功后前端可保持登录态
2. 退出后私有页面无法继续访问

#### B-03 实现找回密码与修改密码

范围：

1. 发送找回验证码
2. 重置密码
3. 登录后修改密码

验收标准：

1. 用户可完成密码重置
2. 旧密码登录失效

#### B-04 实现用户资料页

范围：

1. 获取当前用户资料
2. 修改昵称与简介

### 4.3 阶段 C：经文主链路

#### C-01 导入圣经数据

范围：

1. 建立圣经版本、卷、节表
2. 编写导入脚本

风险提醒：

1. 必须先确认版权和数据来源

验收标准：

1. 可查询任意卷章节经文
2. 数据格式完整准确

#### C-02 实现随机经文生成服务

范围：

1. 支持 `1/7/12/27/39` 节
2. 支持整章生成
3. 生成连续经文引用

验收标准：

1. 返回结果准确
2. 引用格式正确
3. 跨章节补足逻辑正确

#### C-03 实现经文展示页面

范围：

1. 数量选择按钮组
2. 经文卡片展示
3. 重新随机生成

验收标准：

1. 桌面端和移动端都可正常阅读
2. 长经文不会挤压布局

#### C-04 实现“解经”主按钮 UI

范围：

1. 设计显眼的解经按钮
2. 加入悬停或触摸反馈
3. 保持整体审美统一

验收标准：

1. 按钮在经文卡片区域明显可见
2. 动效不会喧宾夺主

#### C-05 实现解经内容服务

范围：

1. 查询缓存解经结果
2. 调用 AI 或结构化资料生成
3. 返回固定结构内容

验收标准：

1. 解经内容包含规定字段
2. 前端可稳定渲染

#### C-06 实现解经内容展示界面

范围：

1. 解经展开区域或独立阅读页
2. 分段标题与长文滚动体验

验收标准：

1. 解经内容可读性良好
2. 能与当前经文形成明确关联

### 4.4 阶段 D：个人灵修能力

#### D-01 实现经文划线

范围：

1. 选中经文文本
2. 记录高亮颜色
3. 保存选区信息

验收标准：

1. 用户能对经文做至少 3 种颜色高亮
2. 刷新后数据能恢复

#### D-02 实现标注解释

范围：

1. 为选中经文片段添加备注
2. 支持编辑和删除

#### D-03 实现感悟草稿自动保存

范围：

1. 感悟编辑器
2. 自动保存草稿
3. 恢复未发布内容

#### D-04 实现感悟发布与可见性控制

范围：

1. 私有感悟发布
2. 公开感悟发布
3. 感悟详情页

验收标准：

1. 私有内容仅本人可见
2. 公开内容可在公开流展示

### 4.5 阶段 E：公共交流能力

#### E-01 实现公开感悟流

范围：

1. 公开感悟列表
2. 感悟详情页
3. 分页加载

#### E-02 实现评论交流

范围：

1. 发布评论
2. 查看评论列表
3. 删除自己的评论

验收标准：

1. 用户可围绕公开感悟正常交流

#### E-03 实现轮次统计

范围：

1. 统计楼主与单个用户之间的有效互动
2. 达标后生成私信解锁记录

风险提醒：

1. “20 轮”的定义需在开发前最终确认

#### E-04 实现私信解锁与私信会话

范围：

1. 查看解锁状态
2. 接受或拒绝解锁
3. 私信会话列表
4. 私信收发

验收标准：

1. 未达标用户不能私信
2. 达标双方可进入私信会话

### 4.6 阶段 F：赞美与平台治理

#### F-01 实现随机赞美播放

范围：

1. 随机一首赞美
2. 播放 / 暂停 / 循环
3. 移动端播放器适配

验收标准：

1. 音频可正常播放
2. 切换经文时播放器不异常中断

#### F-02 实现举报能力

范围：

1. 举报公开感悟
2. 举报评论
3. 举报私信，首版可选

#### F-03 实现管理后台基础能力

范围：

1. 用户列表
2. 举报处理
3. 内容隐藏
4. 赞美资源管理

## 6. 推荐提交粒度

每个任务最好形成独立提交或独立 PR，避免一次改动过大。

建议粒度：

1. 认证链路一个提交组
2. 经文生成与展示一个提交组
3. 解经服务与 UI 一个提交组
4. 标注与感悟一个提交组
5. 评论、私信与轮次统计一个提交组
6. 赞美与后台一个提交组

## 7. 测试建议

### 单元测试优先

1. 注册和登录校验逻辑
2. 随机经文生成逻辑
3. 解经结构化输出映射
4. 私信解锁轮次统计逻辑

### 集成测试优先

1. 注册到登录完整流程
2. 生成经文到查看解经完整流程
3. 发布公开感悟到交流到私信解锁完整流程

### 手工验证重点

1. 经文阅读体验
2. 解经按钮视觉效果
3. 移动端布局
4. 音频播放交互

## 8. 完成定义

一个任务视为完成，必须同时满足：

1. 功能实现完成
2. 自测通过
3. 不破坏既有主链路
4. 具备基本异常处理
5. 文档或注释同步更新

## 9. AI 助理执行建议

后续 AI 助理每次只聚焦一个任务块，执行顺序建议如下：

1. 先完成阶段 A 和阶段 B
2. 再完成阶段 C
3. 然后完成阶段 D 和阶段 E
4. 最后完成阶段 F

避免：

1. 在账号系统未打通前就开始做评论或私信
2. 在圣经数据未导入前就开始做随机经文
3. 在解经结构未固定前就开始做复杂前端渲染
