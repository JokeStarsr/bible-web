# 基督徒线上 Web 服务需求规格说明书

## 1. 文档目的

本文档用于定义一个面向基督徒用户的线上 Web 服务产品需求，供后续 AI 助理或开发团队据此进行系统设计、任务拆解、编码实现、测试与迭代。本文档强调需求清晰、功能完整、交互明确、边界清楚，并尽量提前覆盖后续开发中常见的歧义点。

## 2. 产品概述

### 2.1 产品定位

本产品是一个以“经文随机领受 + 深度解经 + 灵修记录 + 信徒互动 + 赞美播放”为核心的线上 Web 服务。用户注册登录后，可以随机生成指定数量的经文，查看精读式背景解读，记录自己的属灵感悟，并在公开内容中与其他基督徒交流。

### 2.2 产品目标

1. 帮助用户以更容易进入状态的方式随机领受经文。
2. 提供兼具背景、上下文、神学定位与现实应用的精读式解经内容。
3. 让用户能够对经文进行个人化标注、划线与灵修笔记沉淀。
4. 让公开感悟形成有秩序、有边界的交流场景。
5. 通过赞美诗随机播放营造更适合默想和读经的氛围。
6. 在视觉、排版、响应式布局方面提供美观、安静、适合灵修的使用体验。

### 2.3 目标用户

1. 有日常灵修、读经习惯的基督徒。
2. 希望通过随机经文获得每日提醒、安慰、引导的用户。
3. 希望理解经文时代背景、上下文和神学位置的用户。
4. 有表达、分享、交流属灵感悟需求的用户。

## 3. 产品愿景与设计原则

### 3.1 愿景

打造一个既有属灵温度、又有内容深度、同时适合长期使用与沉淀的基督徒线上读经与交流平台。

### 3.2 设计原则

1. 以经文为中心，不喧宾夺主。
2. 视觉安静、庄重、温暖，避免过度商业化。
3. 功能完整但不复杂，降低上手门槛。
4. 解经内容要结构化、可读、可信、可持续扩展。
5. 互动有边界，保护用户隐私与交流秩序。
6. 在桌面端与移动端都保持良好阅读体验。

## 4. 角色定义

### 4.1 游客

1. 可访问首页、产品介绍页、登录页、注册页。
2. 不可使用随机经文、标注、解经、感悟记录、公共交流、私信功能。
3. 可查看部分公开介绍内容，但不可查看用户私有灵修记录。

### 4.2 注册用户

1. 可注册、登录、退出登录、找回密码、修改密码。
2. 可生成随机经文。
3. 可查看“解经”内容。
4. 可对经文进行划线、标注、写感悟。
5. 可选择感悟为私有或公开。
6. 可参与公开感悟下的交流。
7. 当交流满足条件后，可触发私信能力。
8. 可使用随机赞美播放功能。

### 4.3 管理员

1. 管理用户账号状态。
2. 管理公开内容、评论内容、违规内容。
3. 管理私信触发规则的启用状态。
4. 管理经文数据库版本、赞美诗资源、解经内容来源与审核状态。
5. 查看平台统计数据与异常日志。

## 5. 核心使用流程

### 5.1 用户主流程

1. 用户访问网站并完成注册或登录。
2. 进入首页后选择随机生成经文数量：
   - 1 节
   - 7 节
   - 12 节
   - 27 节
   - 39 节
   - 整一章
3. 系统生成经文并以适合阅读的方式展示。
4. 用户可点击显眼的“解经”按钮查看深度解读。
5. 用户可对经文进行划线、标注、解释和写下感悟。
6. 用户可选择将感悟保存为私有或公开。
7. 若为公开，其他用户可阅读并参与交流。
8. 若某条交流累计达到 20 轮以上，系统可解锁双方私信能力。
9. 用户可同时使用随机赞美功能播放赞美诗，辅助默想。

## 6. 功能需求总览

### 6.1 账号系统

必须具备完整账号能力：

1. 注册
2. 登录
3. 退出登录
4. 邮箱或手机号验证
5. 密码找回
6. 重置密码
7. 修改密码
8. 基础个人资料管理

### 6.2 经文随机生成系统

必须支持：

1. 随机生成 1 节经文
2. 随机生成 7 节经文
3. 随机生成 12 节经文
4. 随机生成 27 节经文
5. 随机生成 39 节经文
6. 随机生成整一章经文

### 6.3 解经系统

1. 对当前生成的经文提供背景解读。
2. 提供精读式解经，而不是仅给出简短摘要。
3. 内容必须包含上下文背景。
4. 内容必须说明这段经文在整本圣经中的定位。
5. 内容必须尝试归纳“神借这段经文要告诉世人的真理”。

### 6.4 经文标注与感悟系统

1. 对经文内容进行划线。
2. 对选中的经文片段写下解释。
3. 对整段经文写下个人感悟。
4. 感悟可选择私有可见或公共可见。

### 6.5 公共交流与私信触发

1. 公开感悟可被其他用户查看。
2. 其他用户可围绕公开感悟进行交流。
3. 当某条交流达到 20 轮以上时，系统触发私信能力。

### 6.6 赞美播放系统

1. 随机生成一首赞美诗或赞美音频进行播放。
2. 支持循环播放。
3. 支持暂停播放。

### 6.7 视觉与适配

1. 经文展示背景需要美观、安静、适合灵修。
2. 布局需要响应式适配桌面端、平板端、手机端。
3. 字体大小需要可读、舒适，并支持基础字号调节。

## 7. 详细功能需求

### 7.1 账号系统需求

#### 7.1.1 注册

1. 用户可通过邮箱注册，后续可扩展手机号注册。
2. 注册信息至少包含：
   - 用户名
   - 邮箱
   - 密码
   - 确认密码
3. 需进行邮箱验证码校验。
4. 用户名不可重复，邮箱不可重复。
5. 密码需满足安全要求：
   - 长度不少于 8 位
   - 包含字母和数字
   - 建议支持特殊字符
6. 注册成功后自动跳转至登录页或直接登录。

#### 7.1.2 登录

1. 支持邮箱 + 密码登录。
2. 后续可扩展第三方登录，但首版不是必须。
3. 登录成功后进入用户主页或灵修首页。
4. 支持“记住我”。
5. 登录失败时给出明确错误提示，但不泄露敏感信息。

#### 7.1.3 找回密码

1. 用户可通过邮箱验证码找回密码。
2. 验证通过后允许重置新密码。
3. 重置成功后应使旧登录状态失效。

#### 7.1.4 安全要求

1. 密码必须加密存储，不可明文保存。
2. 接口必须具备基础防暴力破解机制。
3. 验证码、找回密码链接需设置有效期。
4. 登录态应支持过期、刷新、退出失效。

### 7.2 首页与主界面需求

#### 7.2.1 首页目标

登录后的首页应突出三个核心入口：

1. 随机生成经文
2. 点击进入解经
3. 随机赞美播放

#### 7.2.2 布局建议

1. 页面主视觉以经文卡片为核心。
2. 解经按钮应位于经文卡片附近，形成主要行动入口。
3. 赞美播放器应不打断阅读，但可被方便操作。
4. 标注、感悟、交流功能应围绕当前经文展开，而非跳转过远。

### 7.3 随机经文生成功能需求

#### 7.3.1 生成规则

1. 用户点击数量选项后，系统按选项生成对应经文。
2. “1 节、7 节、12 节、27 节、39 节”默认指连续经文段落。
3. 当目标章节剩余节数不足时，系统可跨章节连续补足，或采用智能选段机制。
4. “整一章”必须完整展示某一章全部经文。
5. 需要保证引用格式准确，例如：创世记 1:1-7。

#### 7.3.2 随机范围

首版需支持从整本圣经范围中随机生成，后续可扩展：

1. 仅旧约
2. 仅新约
3. 仅诗篇
4. 仅福音书
5. 按主题随机

说明：以上扩展项首版可不做，但数据结构和接口设计建议保留扩展性。

#### 7.3.3 展示要求

1. 经文应以高可读性的卡片方式展示。
2. 长经文时自动分段，避免整屏密集文字。
3. 展示经文出处。
4. 支持重新随机生成。
5. 支持收藏当前经文，建议作为首版增强功能。

### 7.4 “解经”功能需求

#### 7.4.1 按钮设计要求

“解经”按钮是本产品的关键体验点，必须重点设计。

要求如下：

1. 在视觉上明显突出，成为经文生成后的核心行动按钮。
2. 风格需要庄重、精致、有吸引力，不可过于俗艳。
3. 建议使用光晕、柔和高亮、描边、轻动效或呼吸感设计来提升注意力。
4. 鼠标悬停或触摸时要有明确反馈。
5. 文案可考虑首版使用“开始解经”或“进入精读解经”。

#### 7.4.2 解经内容结构

点击“解经”按钮后，系统应针对当前经文返回结构化内容，至少包含：

1. 经文摘要
2. 历史背景
3. 写作背景
4. 上下文关系
5. 关键词解析
6. 结构观察
7. 在整本圣经中的位置与关联
8. 神学主题或核心真理
9. 神借这段经文对世人的提醒或启示
10. 对当代信徒的实际应用

#### 7.4.3 解经质量要求

1. 解经内容不能只停留在字面解释。
2. 需要兼顾原始背景、作者写作目的、前后文逻辑和圣经整体叙事。
3. 内容风格应温和、清晰、敬虔，不应带攻击性。
4. 避免断章取义。
5. 若使用 AI 生成解经，需考虑后续人工审核机制或可信资料源校验机制。

#### 7.4.4 展示形式

1. 解经内容可展开在当前页面，也可通过抽屉、弹层、独立阅读页展示。
2. 长文应有目录锚点或分段标题。
3. 支持收起与再次打开。
4. 建议支持复制、收藏、导出分享卡片作为增强功能。

### 7.5 经文划线、标注与感悟需求

#### 7.5.1 划线功能

1. 用户可选中经文中的某个词、某句话或某几节经文。
2. 可进行划线高亮。
3. 支持至少 3 种高亮颜色。
4. 划线数据需与当前用户账号绑定。

#### 7.5.2 标注解释功能

1. 用户可针对选中的经文片段写下解释或备注。
2. 标注应显示创建时间。
3. 用户可编辑、删除自己的标注。

#### 7.5.3 感悟记录功能

1. 用户可针对当前经文写整段感悟。
2. 感悟支持自动保存草稿，避免内容丢失。
3. 感悟支持以下可见性：
   - 私有
   - 公开
4. 私有感悟仅本人可见。
5. 公开感悟可被其他登录用户浏览。

### 7.6 公共交流需求

#### 7.6.1 公开内容流转

1. 用户发布公开感悟后，系统生成一条公开内容页或公开讨论区入口。
2. 其他用户可点赞，建议作为增强功能。
3. 其他用户可围绕该感悟回复交流。

#### 7.6.2 交流机制

1. 交流应按时间顺序展示。
2. 需支持楼主与其他用户多轮对话。
3. 需明确“轮”的定义：
   - 建议定义为双方连续往返各一次记为 1 轮
   - 或者每新增一条有效回复记为 1 条互动，再由系统换算是否达标
4. 首版开发前必须最终确认“20 轮”的精确定义，并在代码层固化。

#### 7.6.3 私信触发规则

1. 当某公开交流达到 20 轮以上时，系统触发私信解锁。
2. 私信解锁建议仅针对该对话中的双方用户生效。
3. 系统需提示双方“你们已满足深入交流条件，可开启私信”。
4. 用户可选择是否开启私信，不应强制自动跳转。
5. 私信功能需要具备屏蔽、举报、关闭入口。

#### 7.6.4 内容安全与秩序

1. 公开感悟和交流内容应具备敏感内容审核能力。
2. 管理员需可删除违规内容。
3. 用户需可举报不当内容。
4. 应限制恶意刷屏、重复发言和骚扰行为。

### 7.7 随机赞美播放需求

#### 7.7.1 功能要求

1. 用户可随机播放一首赞美内容。
2. 支持播放、暂停、继续播放。
3. 支持单曲循环。
4. 建议支持切换下一首作为增强功能。

#### 7.7.2 内容来源

1. 首版需明确赞美内容来源是音频、视频还是外部嵌入链接。
2. 若使用第三方资源，必须确保版权合规。
3. 若使用自有音频资源，应支持后台管理上传与维护。

#### 7.7.3 交互要求

1. 播放器不可遮挡主要读经区域。
2. 在移动端需要可折叠或吸附展示。
3. 若用户切换经文，不应强制中断播放，除非用户主动关闭。

## 8. 页面与交互设计需求

### 8.1 整体视觉风格

1. 风格关键词：宁静、圣洁、温暖、精致、现代。
2. 颜色建议采用柔和浅色系、米白、暖灰、深蓝、金色点缀等方案。
3. 避免过于炫目、复杂、游戏化的视觉效果。

### 8.2 经文展示设计

1. 经文区域需有明显阅读焦点。
2. 行高、字距、边距需要针对长文本优化。
3. 背景图或纹理若使用，必须不影响文字可读性。
4. 支持深色模式可作为增强项。

### 8.3 字体与排版

1. 中文字体需清晰、端正、适合长时间阅读。
2. 英文、章节编号、引用信息需有统一排版规范。
3. 支持至少 3 档字号切换：
   - 小
   - 中
   - 大

### 8.4 响应式适配

1. 桌面端需要突出阅读卡片和解经区域。
2. 平板端需要兼顾阅读与操作。
3. 手机端需要单列布局，保证按钮易点按、文字不拥挤。
4. 长内容滚动时，关键操作按钮应易于再次访问。

## 9. 数据需求

### 9.1 圣经经文数据

系统必须具备结构化圣经文本数据，至少包括：

1. 卷名
2. 章
3. 节
4. 经文正文
5. 语言版本标识

说明：

1. 首版需先确定使用哪一种合法可商用或可授权的圣经版本。
2. 若不同版本版权要求不同，必须在立项时明确。

### 9.2 解经数据

解经数据可来自以下模式之一：

1. 预先整理好的结构化解经库
2. 基于权威资料生成的 AI 解经内容
3. 结构化资料 + AI 生成融合模式

建议首版优先采用“结构化资料 + AI 生成融合模式”，以提升质量和覆盖率。

### 9.3 用户数据

至少需要保存：

1. 用户账号信息
2. 登录凭证相关信息
3. 经文生成记录
4. 划线记录
5. 标注内容
6. 感悟内容
7. 公开交流记录
8. 私信解锁状态
9. 赞美播放偏好，建议作为增强项

## 10. 非功能需求

### 10.1 性能要求

1. 首页打开时间应尽量控制在 3 秒以内。
2. 随机生成经文响应应尽量控制在 2 秒以内。
3. 解经内容若为 AI 生成，可采用流式返回或加载状态提示。
4. 长文本页面滚动要流畅。

### 10.2 可用性要求

1. 用户应能在首次使用时快速理解主流程。
2. 关键按钮文案必须清楚。
3. 错误提示要明确、温和、可操作。

### 10.3 安全要求

1. 用户密码与身份凭证安全存储。
2. 所有鉴权接口必须校验权限。
3. 私有感悟严禁被未授权用户访问。
4. 私信内容必须仅对参与双方可见。
5. 管理后台权限需要分级控制。

### 10.4 隐私要求

1. 用户必须能明确知道自己的感悟是“私有”还是“公开”。
2. 默认建议保存为私有，公开需要用户主动确认。
3. 需提供删除感悟、删除标注、关闭私信等能力。

### 10.5 可维护性要求

1. 前后端模块划分清晰。
2. 核心能力需具备可扩展接口。
3. 数据模型应支持后续新增主题经文、灵修计划、收藏夹等功能。

## 11. 推荐技术实现方向

本节不是强制要求，但建议后续 AI 助理优先参考。

### 11.1 前端建议

1. 使用现代 Web 前端框架，例如 React + Next.js。
2. 使用响应式 UI 方案，例如 Tailwind CSS 或成熟组件库。
3. 对经文阅读、划线标注、富文本输入做组件化封装。

### 11.2 后端建议

1. 提供 REST API 或 GraphQL API。
2. 建议使用 Node.js、NestJS、Express，或其他成熟后端框架。
3. 账号、内容、互动、解经、媒体播放等模块分层设计。

### 11.3 数据库建议

1. 关系型数据库用于用户、内容、互动、权限管理。
2. 可考虑 PostgreSQL 或 MySQL。
3. 对全文检索、推荐和日志分析可扩展搜索或缓存服务。

### 11.4 AI 与内容生成建议

1. 解经内容生成应采用明确 Prompt 模板。
2. 生成内容需按固定结构输出，便于前端渲染。
3. 建议保留人工审核入口或内容标记机制。
4. 对涉及神学争议的内容建议增加免责声明与资料来源说明。

## A. 新增功能模块（已落地）

### A.1 希伯来语发音音频

解经内容中的希伯来原文词汇现在支持点击小喇叭图标收听发音。

- 技术方案：浏览器原生 Web Speech API（`speechSynthesis`），语言设置为 `he-IL`。
- 入口：解经页面中所有通过 `HebrewText` 组件渲染的希伯来文。
- 边界处理：不支持语音合成的浏览器会给出提示；连续点击会自动取消前一次朗读。

详细规格见 [`docs/feature-spec-hebrew-maps.md`](docs/feature-spec-hebrew-maps.md)。

### A.2 圣经地图

新增独立“圣经地图”页面，以交互式地图方式展示圣经历史中的重要地点与行进路线。

- 入口：顶部导航栏“圣经地图”。
- 页面路径：`/maps`。
- 核心能力：
  - 路线选择：保罗三次宣教旅程、摩西出埃及路线、亚伯拉罕迁往迦南路线等。
  - 地点搜索：输入中文地名自动定位并展示注解。
  - 地图交互：缩放、平移、标记点击。
  - 地点注解：包含地名、相关经文引用、历史背景与圣经意义。
- 技术方案：`leaflet` + `react-leaflet`，地图数据静态维护在 `frontend/src/data/bibleMaps.ts`。

> 说明：地图中的地点坐标为学术界常用的近似坐标，仅供辅助读经使用，不作为考古或神学定论。

详细规格见 [`docs/feature-spec-hebrew-maps.md`](docs/feature-spec-hebrew-maps.md)。

## 16. 验收标准

### 16.1 功能验收

1. 用户能成功完成注册、登录、找回密码与退出登录。
2. 用户能稳定生成指定数量的随机经文。
3. 用户点击“解经”按钮后能看到结构化深度解读。
4. 用户能对经文划线、标注并保存感悟。
5. 用户能正确设置感悟为私有或公开。
6. 公开感悟能被其他用户浏览并交流。
7. 达到设定条件后，系统能触发私信能力。
8. 随机赞美能正常播放、暂停、循环。

### 16.2 体验验收

1. 页面整体视觉美观、安静、适合灵修。
2. 经文阅读在桌面端和移动端都清晰舒适。
3. “解经”按钮足够醒目，且不破坏整体审美。
4. 交互流程自然，用户不需要复杂学习即可使用。

### 16.3 安全验收

1. 私有内容不泄露。
2. 密码不明文存储。
3. 权限控制有效。
4. 公开区和私信区具备基础内容治理能力。

## 17. 给后续 AI 助理的执行要求

后续 AI 助理在开始开发前，应按以下顺序推进：

1. 先基于本文档输出系统架构方案。
2. 再输出数据库表设计与接口清单。
3. 然后拆分 MVP 开发任务。
4. 优先实现账号系统、经文随机生成、解经展示三大核心链路。
5. 之后实现标注感悟、公开交流、私信触发、赞美播放。
6. 最后完成后台管理、性能优化、内容审核和视觉打磨。
